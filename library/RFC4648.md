Normalised extract (implementation facts)
- Alphabets (exact strings):
  - BASE16 (hex, uppercase canonical): "0123456789ABCDEF"
  - BASE32 (RFC 4648): "ABCDEFGHIJKLMNOPQRSTUVWXYZ234567"
  - BASE32HEX (alternate): "0123456789ABCDEFGHIJKLMNOPQRSTUV"
  - BASE64 (standard): "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/"
  - BASE64URL (URL-safe): "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_"  (same ordering as standard but +/ -> -_)
- Bit grouping and block sizes:
  - Base16: 4 bits per output character (1 input byte -> 2 chars). No padding required.
  - Base32: 5 bits per output character. Process input in 40-bit (5-byte) groups -> 8 output characters. If input length is not a multiple of 5, pad with zero bits on the final partial group and (optionally) add '=' pad characters so the output length is a multiple of 8.
  - Base64: 6 bits per output character. Process input in 24-bit (3-byte) groups -> 4 output characters. If input length is not a multiple of 3, pad with zero bits and append '=' characters so the output length is a multiple of 4.
- Padding rules (canonical RFC behavior):
  - Base32 and Base64: when using the canonical form, output MUST be padded with '=' to reach the full-block output length (8 for Base32, 4 for Base64). Padding may be omitted by applications that use alternate variants (e.g., RFC4648 allows unpadded variants for URLs and filenames).
- Decoding rules (strict vs tolerant):
  - Strict implementations reject characters outside the alphabet and reject incorrect padding placement or unexpected padding characters. Tolerant implementations should strip whitespace and ignore line breaks and optionally accept unpadded inputs for known variants (e.g., base64url without padding).

Reference implementation patterns and pseudo-code
- encode(bytes[], alphabet, bitsPerChar, padChar, pad):
  - accumulator = 0; bits = 0; out = []
  - for each byte b in bytes:
      accumulator = (accumulator << 8) | b
      bits += 8
      while bits >= bitsPerChar:
        bits -= bitsPerChar
        index = (accumulator >> bits) & ((1 << bitsPerChar) - 1)
        out.push(alphabet[index])
  - if bits > 0:
      index = (accumulator << (bitsPerChar - bits)) & ((1 << bitsPerChar) - 1)
      out.push(alphabet[index])
  - if pad:
      while out.length % (blockOutputSize) != 0:
        out.push(padChar)
  - return out.join('')
- decode(str, alphabet, bitsPerChar, padChar):
  - map chars -> value using a lookup table (reject unknown chars)
  - accumulator = 0; bits = 0; outBytes = []
  - for each char c in filteredInput (strip whitespace, ignore padChar):
      accumulator = (accumulator << bitsPerChar) | value(c)
      bits += bitsPerChar
      while bits >= 8:
        bits -= 8
        byte = (accumulator >> bits) & 0xFF
        outBytes.push(byte)
  - return Uint8Array(outBytes)

Implementation API signatures (recommended)
- encodeBase64(input: Uint8Array, urlSafe: boolean = false, pad: boolean = true) -> string
- decodeBase64(input: string, urlSafe: boolean = false) -> Uint8Array
- encodeBase32(input: Uint8Array, variant: 'base32'|'base32hex', pad: boolean = true) -> string
- decodeBase32(input: string, variant) -> Uint8Array

Supplementary details
- Use lookup tables (Uint8Array[256]) for decoding to achieve O(n) performance.
- When using base64url in web contexts, prefer the unpadded variant only when interoperable with consumers that accept it; keep canonical padded form for storage and interchange.
- When interoperating with other libraries, always confirm alphabet order (some libraries swap case or change ordering for base62-like encodings).
