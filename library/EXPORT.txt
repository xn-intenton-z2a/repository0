NORMALISED EXTRACT

1. Export syntax forms
- Declarations: export let name, export const name = value, export function fn(){}, export class C{}
- Export list: export { name1, name2 }
- Aliased export: export { variable as name }
- Default export: export default expression | function | class
- Aggregation: export * from "module-name"; export { name } from "module-name"; export * as ns from "module-name"

2. Named vs default exports
- Named exports: multiple per module; imported with exact names (can alias via import { name as alias }).
- Default export: exactly one per module; importable with any local identifier via import localName from 'module'.
- Cannot have duplicate exported names or more than one default; duplicate causes SyntaxError at module evaluation.

3. Re-exporting (export from)
- export { x } from 'mod' forwards named export x without binding it in the current module scope.
- export * from 'mod' re-exports all named exports (not default) and will conflict if multiple sources export same name.
- export { default as name } from 'mod' re-exports module default as named export.
- export { default } from './data.json' with { type: 'json' } supports import attributes where applicable.

4. Constraints and runtime effects
- Export declarations are module-level only and are not subject to temporal dead zone; export { x } can precede declaration of x.
- Use of export with using/await using is syntactically disallowed; exporting a disposed resource is discouraged.
- Re-export conflicts cause SyntaxError when importing conflicting names.

SUPPLEMENTARY DETAILS

Implementation notes for modules:
- Always use module-aware bundlers/runtimes; include script type="module" in browsers for module evaluation.
- When building barrels (aggregator modules), avoid exporting same names from multiple aggregated modules.
- For JSON/module import attributes, follow host runtime support (browser/node with import attributes limited to type: 'json').

REFERENCE DETAILS

Precise grammar / allowed forms (ECMAScript):
- export Declaration
- export VariableStatement
- export FunctionDeclaration
- export ClassDeclaration
- export default AssignmentExpression
- export NamedExports
- export ExportFrom (exportClause from ModuleSpecifier)

Best practices:
- Prefer named exports for libraries that expose multiple utilities; reserve default when module represents single primary value.
- Use explicit re-export naming to avoid accidental conflicts: export { default as main } from './sub'
- When re-exporting many modules, use a build step to detect duplicate named exports.
