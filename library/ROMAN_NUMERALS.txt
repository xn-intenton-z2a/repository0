2. CANONICAL RULES AND LIMITS
- Use additive notation for repeated symbols (e.g., II = 2, XXX = 30, CCC = 300).
- Use subtractive notation only for specific pairs: I before V or X (IV=4, IX=9); X before L or C (XL=40, XC=90); C before D or M (CD=400, CM=900).
- The only subtractive combinations in standard (modern/classical) form: IV, IX, XL, XC, CD, CM.
- Do not repeat a symbol more than three times in succession in canonical form (III is allowed; IIII is non-standard for canonical representation).
- Canonical representable range using standard modern notation: 1..3999 (largest canonical numeral: 3999 = MMMCMXCIX).
- No zero; negative values unsupported in Roman numeral system.

4. VALIDATION (REGULAR EXPRESSION)
- Strict canonical validation regex (accepts uppercase and enforces subtractive rules and repetition limits):
  ^M{0,3}(CM|CD|D?C{0,3})(XC|XL|L?X{0,3})(IX|IV|V?I{0,3})$
- Preprocess: trim whitespace and convert to uppercase before matching.
- If regex fails, treat the input as invalid canonical Roman numeral (reject or normalize only if non-standard modes allowed).

5. DECIMAL → ROMAN: ALGORITHM (step-by-step)
Preconditions: input n must be integer; canonical range 1..3999.
1. Validate type: if input is not integer, raise TypeError (or return error according to API contract).
2. Validate range: if n < 1 or n > 3999, raise RangeError (or return error).
3. Prepare an ordered array of (value, symbol) pairs as in Mapping Table, highest to lowest.
4. Initialize result string r = empty and remaining rem = n.
5. For each pair (v, s) in the ordered array:
   a. While rem >= v:
      i. Append s to r.
      ii. Subtract v from rem (rem -= v).
6. After iterating all pairs, rem must be 0 and r is the canonical Roman numeral.
7. Return r.
Notes: iterative subtraction ensures that subtractive pairs (900, 400, 90, 40, 9, 4) are used where appropriate because these pairs appear before their additive counterparts in the ordered array.

6. ROMAN → DECIMAL: ALGORITHM (step-by-step)
Preconditions: input must be a string; canonical validation recommended.
1. Validate type: if input is not a string, raise TypeError (or return error).
2. Normalize input: s = input.trim().toUpperCase().
3. Validate canonical form using regex in section 4. If validation fails, raise TypeError (or return error) unless non-standard modes are permitted.
4. Prepare a mapping table from single symbols to integer values: {I:1, V:5, X:10, L:50, C:100, D:500, M:1000}.
5. Initialize total = 0 and index i = 0.
6. Iterate while i < s.length:
   a. Let cur = valueOf(s[i]) and next = valueOf(s[i+1]) if i+1 < s.length else 0.
   b. If cur < next: add (next - cur) to total and increment i by 2 (skip the next character).
   c. Else: add cur to total and increment i by 1.
7. Return total.
Notes: This lookahead algorithm handles canonical subtractive pairs because the validation step ensures only valid subtractive patterns occur; no need to hard-code subtractive pair list in conversion loop.

7. NON-STANDARD VARIANTS AND EDGE CASES (implementation options and effects)
- Additive forms: allow IIII, XXXX, CCCC, VIIII etc. These are historically attested but non-canonical; permitting them requires relaxed validation and normalization rules.
- Alternate subtractive forms: historical examples include IIX for 8 or IC for 99; these are non-standard and should be rejected by canonical validation.
- Clock-face exception: many clocks render 4 as IIII; to support this, allow an option (allowClockForm=true) that accepts IIII and maps to 4 when decoding, and outputs IIII when encoding if requested.
- Large numbers (>3999): various marking systems (overlines, vinculum) exist historically. Implementations may either reject >3999 or accept an extended format (e.g., allow overline notation or multiplier tokens) if explicitly required.
- Zero and negatives: Roman numerals have no zero; define API behavior (throw error or return null) for 0 or negative numbers.

8. SUPPLEMENTARY DETAILS (constraints, performance, usage notes)
- Complexity: Decimal→Roman is O(k) where k is the number of symbol additions (worst case proportional to magnitude, but bounded since canonical max is 3999). Roman→Decimal is O(n) in the length of the string (max length limited by canonical rules).
- Input sanitization: trim, uppercase; reject empty string as invalid.
- Error handling: prefer throwing typed exceptions (TypeError for wrong type, RangeError or ValueError for out-of-range) to simplify caller handling.
- Internationalization: Roman numerals are locale-independent; no character encoding beyond ASCII required.
- Security: no external dependencies required for conversion; avoid eval or dynamic parsing of patterns.

9. REFERENCE API SPECIFICATIONS AND METHOD SIGNATURES
- Function: toRoman
  Signature: toRoman(n: integer) -> string
  Preconditions: n is an integer; 1 <= n <= 3999
  Errors: throws TypeError if n is not integer; throws RangeError if n < 1 or n > 3999
  Behaviour: returns canonical Roman numeral string (uppercase) using mapping and algorithm in sections 3 and 5.
- Function: fromRoman
  Signature: fromRoman(s: string) -> integer
  Preconditions: s is a string representing a canonical Roman numeral (per regex in section 4)
  Errors: throws TypeError if s is not string or does not match canonical regex
  Behaviour: returns integer value of canonical Roman numeral string using algorithm in section 6.

11. TROUBLESHOOTING AND COMMON FAILURES
- Failure: Input integer 0 or negative.
  Remedy: Enforce precondition; if user input may be zero, map to explicit error message: "toRoman: value must be in range 1..3999".
- Failure: fromRoman rejects valid-looking but non-canonical forms (e.g., IIII or IIX).
  Remedy: Provide an optional permissive mode that normalizes common non-canonical forms before parsing, or document strict canonical behaviour clearly.
- Failure: Mixed-case or extra whitespace causes unexpected rejection.
  Remedy: Always trim() and toUpperCase() before validation.
- Failure: Unicode-looking characters (e.g., fullwidth I) used.
  Remedy: Normalize unicode NFKC or validate characters against ASCII set I,V,X,L,C,D,M explicitly.
