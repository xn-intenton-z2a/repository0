1. Numeral System Overview
Roman numerals represent integers using combinations of letters from the Latin alphabet. Numbers are formed by adding symbol values, except where a smaller value precedes a larger value to indicate subtraction. The system commonly used for modern implementation supports symbols for units up to 1000 and uses subtractive notation for 4, 9, 40, 90, 400, 900.

2. Core Symbols and Values
I = 1
V = 5
X = 10
L = 50
C = 100
D = 500
M = 1000

Extended forms (Unicode number forms for Roman numerals):
U+2160..U+216F contain uppercase Roman numeral glyphs for some numbers; U+2170..U+217F contain lowercase forms. These are typographic glyphs, not new numeric system rules.

3. Construction Rules (Additive/Subtractive)
- Additive: Symbols of equal or decreasing value placed left-to-right are summed. Example: XVI = 10 + 5 + 1 = 16.
- Subtractive pairs: Only the following smaller-before-larger pairs are valid:
  I before V or X (IV=4, IX=9)
  X before L or C (XL=40, XC=90)
  C before D or M (CD=400, CM=900)
- Prohibition rules:
  Do not repeat V, L, or D consecutively.
  I, X, C, and M may be repeated up to three times in succession (III, XXX, CCC, MMM).
  Do not use subtractive notation with more than one smaller symbol (e.g., IIX invalid).

4. Conversion Algorithms
Integer to Roman (deterministic greedy algorithm):
- Maintain ordered pairs list of numeral strings and integer values (descending): M=1000, CM=900, D=500, CD=400, C=100, XC=90, L=50, XL=40, X=10, IX=9, V=5, IV=4, I=1.
- For each pair (symbol, value): while input >= value: append symbol to output; input -= value.
- Result is canonical Roman numeral using standard subtractive rules.

Roman to Integer (parser algorithm):
- Normalize input to uppercase and validate characters only in {I,V,X,L,C,D,M} or Unicode roman glyphs mapped to these.
- Iterate left-to-right accumulating value: for each symbol, lookahead to next symbol; if current value < next value, subtract current from total; else add current.
- Validation steps: enforce repetition rules and allowed subtractive pairs; reject invalid sequences (e.g., IL, VX, IIX, VV).

Complexity: both algorithms operate in O(n) time where n is number of symbols; integer→Roman loops over constant list of value pairs (O(1) per digit magnitude), Roman→integer is a single pass.

5. Edge Cases and Limits
- Typical canonical implementations restrict input domain to 1..3999 because standard notation uses M (1000) repeated up to three times for 3000 and no standard symbol for 5000+. Some libraries support overline notation for multiples of 1000 (not covered by basic implementation).
- Zero and negative numbers: no Roman numeral representation; implementations should reject or throw on values <= 0.
- Non-canonical forms: some historical forms exist (e.g., IIII on clocks) — decide whether to accept them; default behaviour: accept only canonical forms.

6. Unicode and Glyphs
- Unicode glyph block U+2160..U+216F contains precomposed Roman numeral characters for values I..L.. and beyond; treat these as single-character equivalents when parsing text but map them back to ASCII-letter sequences for algorithmic conversion.
- Normalization: map Unicode Roman glyphs to ASCII letters before applying parsing rules to ensure consistent validation and conversion.

8. Implementation Patterns and Best Practices
- Use an ordered immutable mapping of symbol→value pairs including subtractive pairs for deterministic conversion.
- For integer→Roman, use greedy subtraction over the descending mapping list to guarantee canonical output.
- For Roman→integer, implement a single-pass parser with lookahead and strict validation of allowed subtractive pairs and repetition counts.
- Normalize inputs: trim whitespace, uppercase ASCII letters, map Unicode Roman glyphs to ASCII equivalents, verify ASCII-only if expecting plain ASCII API.
- Provide explicit error messages for out-of-range integers, invalid characters, invalid subtractive pairs, and repetition violations.
- Expose option flags if needed: allowNonCanonical (boolean) to accept historical variants; maxUpperLimit (integer) to support overline/extended notation when implemented.

9. Troubleshooting and Test Cases
Essential unit tests (minimal set):
- Integer→Roman: 1→I, 4→IV, 9→IX, 14→XIV, 44→XLIV, 944→CMXLIV, 1987→MCMLXXXVII, 3999→MMMCMXCIX.
- Roman→Integer: I→1, IV→4, IX→9, XL→40, CM→900, MMMCMXCIX→3999.
- Invalid inputs: 0, -1, 4000 (out-of-range), empty string, "IIII", "VX", "IL" should all produce validation errors.
- Unicode glyph acceptance: U+2160 (Roman Numeral One) parsed as I.
Troubleshooting steps:
- If conversion returns unexpected lower value, check validation of subtractive pairs and correct lookahead logic.
- If integer→Roman produces non-canonical forms, ensure mapping list includes subtractive pairs at correct positions (e.g., CM before D and C).
- For parsing failures with Unicode, ensure normalization mapping is applied before validation.

10. Supplementary Technical Specifications
- Data structures: use an ordered array of tuples [("M",1000), ("CM",900), ...] for greedy conversion; use a map/dictionary for symbol→value lookups in parser.
- Memory: negligible; algorithms are streaming in nature.
- Concurrency: functions are pure and stateless; safe for concurrent calls.
- Error handling: prefer throwing typed errors (RangeError for out-of-range, TypeError for invalid type) or return result objects {ok: boolean, value?: number, error?: string} depending on API design.

11. Reference Details (Exact APIs / Signatures / Config)
API signatures (canonical):
function toRoman(n: number): string
  - Input: n: integer, 1 <= n <= 3999
  - Output: canonical Roman numeral string
  - Errors: throws RangeError if n < 1 or n > 3999; throws TypeError if n not integer

function fromRoman(s: string): number
  - Input: s: string containing ASCII Roman numeral or mapped Unicode glyphs
  - Output: integer value
  - Errors: throws SyntaxError if s contains invalid characters or non-canonical sequences

function validateRoman(s: string): boolean
  - Input: s: string
  - Output: true if s is canonical Roman numeral in accepted domain; false otherwise

Configuration options (for instantiable implementations):
- allowNonCanonical: boolean (default false)
  - If true, parser accepts certain historical variants (e.g., IIII) and returns best-effort integer value.
- maxValue: integer (default 3999)
  - Upper inclusive bound for toRoman; values above must be handled via extended notation or rejected.
- unicodeMapping: boolean (default true)
  - If true, map Unicode Roman numeral glyphs to ASCII letters before parsing.
