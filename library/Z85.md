Normalised extract
Z85 (ZeroMQ RFC 32) is a variant of base-85 encoding tailored for safe inclusion in source code and command lines. It encodes 4-byte groups to 5 printable ASCII characters. Z85 requires binary frame length divisible by 4 and produces string lengths divisible by 5; applications must pad or frame as needed. The specification defines an exact 85-character alphabet chosen for readability and safety.

Technical details
- Bits per char: log2(85) ≈ 6.409 bits/char
- Frame constraint: binary length % 4 == 0; encoded length % 5 == 0
- Alphabet (85 characters) in order (index 0..84):
  0..9        : 0123456789
  10..35      : abcdefghijklmnopqrst (continues through 't')
  36..61      : uvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ (continues)
  remaining   : .-:+=^!/*?&<>()[ ]{}@%$#  (final printable characters per spec)

Canonical Z85 alphabet (single-line):
0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ.-:+=^!/*?&<>()[]{}@%$#

Encoding algorithm (Uint8Array -> string)
- Input must be divisible by 4; application pads if necessary
- For each 4-byte block: value = (b0<<24)|(b1<<16)|(b2<<8)|b3 (big-endian unsigned 32-bit)
- For i = 4 down to 0: divisor = 85^i; digit = Math.floor(value / divisor); value -= digit * divisor; append alphabet[digit]

Decoding algorithm (string -> Uint8Array)
- Input length must be divisible by 5; application strips or validates framing
- For each 5-char block: value = 0; for each char c: digit = alphabetIndex(c); value = value*85 + digit; output bytes: [(value>>24)&0xFF, (value>>16)&0xFF, (value>>8)&0xFF, value&0xFF]

Supplementary details
- Z85 removes ambiguous or unsafe characters and ensures readability in source code contexts
- Implementations must provide explicit padding or length framing since Z85 itself forbids partial final blocks
- Reference implementation available in ZeroMQ RFC sources; follow exact alphabet ordering to ensure interoperability

Reference details (method signatures)
- encodeZ85(bytes: Uint8Array) => string
- decodeZ85(text: string) => Uint8Array
- Note: both functions should throw on length mismatch (not divisible by 4/5) unless caller provides padding strategy
