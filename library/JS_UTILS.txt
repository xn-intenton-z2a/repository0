1. Modules and export/import
- Export syntax (ECMAScript modules):
  - Export declarations: export let name1, name2; export const name1 = 1; export function f() {}
  - Export list: export { name1, name2 } or export { variable1 as name1 }
  - Default exports: export default expression; export default function() {} (only one default per module)
  - Aggregating/re-export: export * from "module-name"; export { name1 } from "module-name"; export * as ns from "module-name"
  - Re-export rules: export * re-exports only named exports (not default). If two wildcard exports define the same name, neither is re-exported and importing that name triggers a SyntaxError.
- Import syntax (for completeness): import { a, b } from "./mod.js"; import defaultExport from "./mod.js"; imported bindings are read-only views; their value can only be reassigned by the exporter.
- Import maps (browser): JSON object in <script type="importmap"> with "imports" mapping specifiers to absolute/relative URLs. Import maps apply only to documents; they do not apply automatically to workers.
- File extension and module parsing:
  - .mjs: treated as ECMAScript module
  - .js: treated according to nearest package.json "type" field ("module" or "commonjs").
- Important runtime failures:
  - Duplicate export names or multiple default exports => SyntaxError.
  - Serving modules with incorrect Content-Type (not text/javascript) => browser MIME type error.

2. Node.js module resolution and system
- Two systems: CommonJS (require, module.exports, exports) and ECMAScript modules (import/export, dynamic import()).
- Determining module type for .js files:
  - If nearest package.json has "type": "module" => parse as ESM
  - If "type": "commonjs" => parse as CommonJS
  - If no package.json => default CommonJS for .js
- require() lookup algorithm (high level):
  1. If X is core module => return core module
  2. If X is path (./ ../ /) => LOAD_AS_FILE then LOAD_AS_DIRECTORY
  3. If X begins with '#' => LOAD_PACKAGE_IMPORTS
  4. Else LOAD_PACKAGE_SELF then LOAD_NODE_MODULES then throw not found
- LOAD_AS_FILE tries exact filename, then X.js, X.json, X.node and uses package.json "type" to decide parsing for .js
- LOAD_AS_DIRECTORY tries package.json main field, then index.js/index.json/index.node
- NODE_MODULES_PATHS builds node_modules search paths by walking parent directories and appending node_modules
- Package fields with direct effects:
  - "type": "module" | "commonjs" — affects how .js is parsed
  - "main": entrypoint for LOAD_AS_DIRECTORY
  - "exports": conditional exports mapping for package import resolution
  - "imports": internal package import maps for '#'-prefixed specifiers
- require() can synchronously load ESM under strict conditions (no top-level await; file extension/type rules); errors: ERR_REQUIRE_ASYNC_MODULE when encountering top-level await
- Interop notes: require(esm) returns module namespace object; default export appears as .default; __esModule may be added by tooling for compatibility

3. Array.from
- Signature: Array.from(items[, mapFn[, thisArg]]) -> Array
- Parameters:
  - items: iterable or array-like (must have length property for array-like)
  - mapFn (optional): function(element, index) applied to each produced element; called with thisArg if supplied
  - thisArg (optional): value to use as this when executing mapFn
- Behavior:
  - Converts iterable or array-like to a new Array instance (never sparse; missing indices become undefined)
  - If called on subclass (this value as constructor), returns instance of that subclass
  - Equivalent to Array.from(items).map(mapFn, thisArg) but without intermediate array allocation and mapFn receives only element and index
- Use-cases: converting NodeList/Set/Map values to arrays, generating ranges: Array.from({length: N}, (_, i) => i)
- Spec reference: ECMAScript Array.from section (ECMA-262)

4. Number.isInteger
- Signature: Number.isInteger(value) -> boolean
- Semantics:
  - Returns true if type is Number and it's an integer (no fractional part)
  - Returns false for NaN, Infinity, non-number types, and for numbers with non-zero fractional part
  - Will return true for floating-point values that represent integers due to IEEE-754 precision (e.g., 5.0000000000000001 may be represented as 5)
- Edge cases: large magnitudes near Number.MAX_SAFE_INTEGER may suffer precision loss causing surprising true/false values
- Spec reference: ECMAScript Number.isInteger

5. RangeError
- Definition: RangeError is an Error subclass thrown when a value is outside allowed range for an operation
- Common triggers:
  - Array constructor with illegal length
  - Number.prototype.toExponential/toFixed/toPrecision with invalid arguments
  - String.prototype.normalize with invalid form
- Constructor: RangeError([message]) -> RangeError instance with name = "RangeError"
- Instance properties: inherits from Error (message, stack, name)
- Usage pattern: validate inputs and throw new RangeError("message") when value out-of-range; catch using instanceof RangeError to handle specifically

6. Creating Node.js modules (npm guidelines)
- Steps to publish a module:
  1. npm init (or npm init --scope=@scope-name) to create package.json; required fields: name, version, main
  2. Implement entry file as specified by "main" (exports.<fn> = function() { ... } or module.exports = ... )
  3. Test by installing in an external test directory and require/import it; run node test.js
  4. Publish with npm publish (requires 2FA or appropriate access tokens)
- package.json fields with direct effects:
  - name: package identifier used in install
  - version: semantic version
  - main: entrypoint for require() when requiring package by name
  - type: affects .js parsing (module vs commonjs)
  - exports/imports: control package entry points and internal specifiers
- Best-practices:
  - Follow semver for versioning
  - Provide both ESM and CommonJS entry points if supporting both (use exports map)
  - Test install and require path from an external folder before publishing

7. FizzBuzz algorithmic rules and minimal implementation pattern
- Rules:
  - For each integer n in sequence starting at 1:
    - If n % 15 === 0 => output "FizzBuzz"
    - Else if n % 3 === 0 => output "Fizz"
    - Else if n % 5 === 0 => output "Buzz"
    - Else output n
- Minimal implementation pattern (language-agnostic): loop with conditional checks in order (15, then 3, then 5) to avoid incorrect replacements
- Use-cases: simple unit tests, interview screening, demonstration of control flow and modularization

SUPPLEMENTARY DETAILS (SPECIFICATIONS, IMPLEMENTATION NOTES)
- Module filenames and server MIME:
  - Serve module files with Content-Type: text/javascript; mismatched MIME results in browser refusal to execute
  - If using .mjs ensure server supports .mjs MIME mapping or use build step to rename to .js with appropriate headers
- ESM/CJS coexistence strategies:
  - Provide "type": "module" and dual entry points via exports map for conditional exports targeting require vs import
  - Use interop patterns: place default export on .default property for CJS consumers or expose module.exports via export { default as 'module.exports' }
- Array.from performance notes:
  - Avoid creating unnecessary intermediate arrays; prefer Array.from(items, mapFn) over Array.from(items).map(mapFn) when mapping
  - For typed arrays, mapFn can avoid truncation issues of intermediate arrays
- Error handling and validation patterns:
  - Validate range-sensitive parameters early and throw RangeError with clear message; document expected numeric ranges
  - For API surface, prefer throwing RangeError for numeric bounds and TypeError for wrong types

- Node.js require algorithm (pseudocode exact sequence):
  - require(X) from path Y:
    1. If X is core module => return core module
    2. If X begins with '/' => set Y root
    3. If X is '.' or starts with './' or '../' => LOAD_AS_FILE(Y+X) then LOAD_AS_DIRECTORY(Y+X) then throw
    4. If X begins with '#' => LOAD_PACKAGE_IMPORTS(X, dirname(Y))
    5. LOAD_PACKAGE_SELF(X, dirname(Y))
    6. LOAD_NODE_MODULES(X, dirname(Y))
    7. THROW not found
  - LOAD_AS_FILE tries: exact path, path.js, path.json, path.node; uses package.json "type" to determine .js parsing; detects ESM/CJS via syntax when necessary
  - NODE_MODULES_PATHS constructs search list by walking up directories: for each ancestor add ancestor/node_modules

  - "type": "module" | "commonjs" (affects .js parsing in Node.js)
  - "main": string (file path used when requiring directory)
  - "exports": object or string mapping subpath request to file(s); supports conditional exports for different environments
  - "imports": object mapping internal specifiers beginning with '#' to paths within package
- npm publish requirements:
  - For publishing, 2FA may be required or a granular access token with bypass 2FA enabled
  - For scoped public packages use: npm publish --access public

BEST PRACTICES & TROUBLESHOOTING
- Browser modules:
  - Ensure server sends Content-Type: text/javascript; if seeing MIME type errors, confirm server mapping for .mjs/.js
  - Use <script type="module"> for entry point modules
- Node interoperability:
  - For packages supporting both CJS and ESM, use exports map and provide separate entry points; test both import() and require()
  - To avoid ERR_REQUIRE_ASYNC_MODULE, do not call require() on modules containing top-level await; instead use dynamic import()
- Debugging module resolution:
  - Use require.resolve(<specifier>) to see resolved filename used by require()
  - When package import fails, inspect node_modules lookup paths (NODE_MODULES_PATHS) and package.json "exports" and "imports"
- Publishing issues:
  - If npm publish fails, verify version bump, package name uniqueness, and 2FA/token requirements
