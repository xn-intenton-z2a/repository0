Overview
This document extracts the technical UUID byte-level specification from RFC 4122 required for deterministic encoding/decoding and transformations used by the mission.

Canonical string format
- A UUID is represented as 32 hexadecimal characters displayed in five groups separated by hyphens: 8-4-4-4-12 (for example: 01234567-89ab-cdef-0123-456789abcdef).
- The canonical (text) representation uses lower-case hex (recommended normalization) and hyphens at positions 8,13,18 and 23.

Byte layout (octet indices)
When mapping the canonical string to a 16-byte array (Uint8Array[16]) the mapping is:
- bytes 0..3  (4 bytes) => time_low  => first 8 hex digits
- bytes 4..5  (2 bytes) => time_mid  => next 4 hex digits
- bytes 6..7  (2 bytes) => time_hi_and_version => next 4 hex digits
- byte  8     (1 byte)  => clock_seq_hi_and_reserved => next 2 hex digits
- byte  9     (1 byte)  => clock_seq_low => next 2 hex digits
- bytes 10..15 (6 bytes) => node => last 12 hex digits

Notes on endianness/ordering
- Each field is presented in network byte order (big-endian) within the canonical string; the hex sequence maps directly to bytes in the order shown above.

Version and Variant bit extraction and setting
- Version field:
  - Located in the 4 most significant bits of octet 6 (zero-based index). Implementation:
    - version = (bytes[6] & 0xF0) >> 4
  - To set a version V (0..15): bytes[6] = (bytes[6] & 0x0F) | ((V & 0x0F) << 4)
- Variant field (RFC 4122 variant):
  - Encoded in the most significant bits of octet 8 (clock_seq_hi_and_reserved).
  - RFC 4122 variant corresponds to binary pattern 10xxxxxx (i.e. (octet8 & 0xC0) == 0x80)
  - To set RFC-4122 variant: bytes[8] = (bytes[8] & 0x3F) | 0x80
  - To test RFC-4122 variant: ((bytes[8] & 0xC0) == 0x80)

Parsing and serialisation (hex string <-> Uint8Array)
- String to bytes (strict canonical input allowed with or without hyphens):
  1. Remove hyphens. Verify resulting string length == 32 and contains only hex [0-9A-Fa-f].
  2. For i from 0..15: bytes[i] = parseInt(hex.substr(i*2, 2), 16).
- Bytes to string (canonical):
  1. Produce hex pairs for bytes[0]..bytes[15] in order.
  2. Insert hyphens after 8, 12, 16 and 20 hex characters to form 8-4-4-4-12 grouping.
  3. Lowercase the hex digits for normalization.

Validation checks
- Input hex length must be 32 (after removing hyphens).
- When round-tripping or accepting external UUIDs, confirm that ((bytes[8] & 0xC0) == 0x80) for RFC-4122 variant unless intentionally supporting other variants.
- When expecting a specific version (e.g., v7), check ((bytes[6] & 0xF0) >> 4) == expectedVersion.

Reference details (masks, bit operations, examples)
- Constants:
  - UUID_BYTE_LENGTH = 16
  - VERSION_MASK = 0xF0
  - VERSION_CLEAR = 0x0F
  - VARIANT_MASK = 0xC0
  - VARIANT_RFC4122 = 0x80
  - VARIANT_CLEAR = 0x3F
- Example: set v7 on bytes (JavaScript Uint8Array semantics):
  - // set version 7
  - bytes[6] = (bytes[6] & 0x0F) | (0x07 << 4)
  - // set RFC-4122 variant
  - bytes[8] = (bytes[8] & 0x3F) | 0x80
- Example: extract version and variant
  - const version = (bytes[6] & 0xF0) >> 4
  - const isRFC4122 = (bytes[8] & 0xC0) === 0x80

Shorthand for mission (UUID -> bytes -> encode -> reverse)
- Given a canonical UUID string:
  1. Strip hyphens and parse into a Uint8Array[16] as above.
  2. Pass the raw bytes to the encoding routine (encode(Uint8Array)).
  3. If mission requires reversed output, reverse the encoded string bytes/characters as defined by the library's reversal rule.

Supplementary details
- The RFC describes multiple UUID versions; v1 uses time and node, v4 is random, v6/v7 are newer variants (v7 is defined in a separate draft). RFC 4122 remains the authoritative definition for layout, versions, and variant semantics for UUIDs up to v5.
- For v7-specific bit assignment consult the draft-peabody-dispatch-new-uuid-format source included in project SOURCES.md; treat v7 as an application-level extension with the same 16-byte container and RFC variant bits.
