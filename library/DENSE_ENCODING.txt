Technical content (direct, implementable)

Binary-to-text encoding: purpose and constraints
- Objective: map arbitrary sequences of 8-bit octets to printable text characters safe for transport over channels that are not 8-bit-clean, and to avoid delimiter collisions and control characters. Implementations must consider: safe printable set (commonly printable ASCII 33–126), max line lengths imposed by transports, and whether whitespace and CRLF are permitted or must be preserved.
- Efficiency metric: bits-in / bits-out = log2(n)/8 where n is alphabet size.

Base64 (RFC 4648 canonical rules and alphabet)
- Alphabet (value -> char): 0:A 1:B 2:C 3:D 4:E 5:F 6:G 7:H 8:I 9:J 10:K 11:L 12:M 13:N 14:O 15:P 16:Q 17:R 18:S 19:T 20:U 21:V 22:W 23:X 24:Y 25:Z 26:a 27:b 28:c 29:d 30:e 31:f 32:g 33:h 34:i 35:j 36:k 37:l 38:m 39:n 40:o 41:p 42:q 43:r 44:s 45:t 46:u 47:v 48:w 49:x 50:y 51:z 52:0 53:1 54:2 55:3 56:4 57:5 58:6 59:7 60:8 61:9 62:+ 63:/
- Base64url: same mapping except 62:'-' and 63:'_' and padding '=' optional per referencing spec (use '=' unless spec permits omission).
- Encoding algorithm (canonical): group input into 24-bit blocks (3 bytes), split into four 6-bit values, map each 6-bit value to alphabet char. If remaining bytes < 3, pad with zero bits on the right then emit output characters and append '=' padding: 2 leftover bytes -> 3 chars + '='; 1 leftover byte -> 2 chars + '=='. Encoders MUST set pad bits to zero to ensure canonical encoding.
- Decoding: convert each encoded character to 6-bit value, concatenate groups of 6 bits, drop pad bits as signalled by '=' count. Decoders MUST reject characters outside the alphabet unless the referring spec explicitly allows tolerance (MIME allows ignoring non-alphabet characters like CRLF). RFC 4648: Implementations MUST include appropriate padding unless the referring spec excludes it.
- Line-feeds: Implementations MUST NOT insert line feeds into base-encoded data unless the higher-level specification explicitly requires fixed line length (RFC 4648 forbids automatic line wrapping).

Base32 and Base16 (canonical rules)
- Base32: 5-bit groups, process 40-bit blocks (5 bytes) -> 8 output chars. Padding rules: final quanta may emit between 2 and 8 chars with '=' padding to reach 8-character multiple. Alphabet uses A–Z and 2–7. Bits ordering: most-significant bit first.
- Base16: hex encoding (nibble per char). No padding required.

Ascii85 / Base85 (Adobe / btoa variant)
- Encode 4 input bytes (32 bits, big-endian) as a single 32-bit integer, convert to 5 base-85 digits by repeated division by 85, then add 33 to each digit to map into printable ASCII range 33 ('!') through 117 ('u').
- Special-case optimization: a 32-bit zero block encoded as the single character 'z' (Adobe Ascii85) to compress long zero runs; 'z' is only valid when standing in place of exactly 5 output characters for an all-zero 4-byte input block.
- Final partial block: pad input with up to 3 NUL bytes, encode to 5 chars, then drop trailing output characters corresponding to the padding (Adobe pads with NUL for encoding and uses 'u' for padded decoding as per spec).
- Adobe delimiters: optional prefix '<~' and terminator '~>' for enclosed streams.

basE91
- Variable-width encoding using a 91-character alphabet producing efficiency between ~14% and 23% overhead. Implementation converts streams into variable-length output symbols; refer to basE91 spec for exact integer packing and symbol table. Implementation notes: basE91 is more efficient than base64 for many binary distributions; decoder must implement symbol table and variable-bit assembly.

UUID (RFC 4122 and RFC 9562 updates; variant-1 canonical layout)
- UUID is 128 bits. Variant field determines layout interpretation; most common is variant 1 (DCE/RFC style) where the variant occupies the most significant bits of the 9th byte.
- Version field: 4-bit value in the high 4 bits of the 7th byte (text form: digit after second hyphen).
- Version 1 layout (RFC-style) byte fields and sizes:
  - time_low: 4 bytes (low 32 bits of timestamp)
  - time_mid: 2 bytes (middle 16 bits of timestamp)
  - time_hi_and_version: 2 bytes (high 12 bits of timestamp + 4-bit version)
  - clock_seq_hi_and_reserved + clock_seq_low: 2 bytes (variant in high bits + 14-bit clock sequence)
  - node: 6 bytes (48-bit node id MAC or random; if random, set least-significant bit of first octet to 1 to indicate non-MAC)
- Version 4: random-based — implementation should set the 4-bit version field to 4, and the two MSB of clock_seq_hi_and_reserved to binary 10 (variant). Random bit budget for v4 variant-1 is 122 random bits.
- Version 7: layout: 48-bit big-endian Unix epoch milliseconds, 4-bit version=7, 12 bits for extra precision/monotonicity, 2-bit variant=10, 62 bits of randomness/monotonicity. Use for time-ordered lexicographic-sortable IDs.
- Nil UUID: all zeros. Max UUID: all ones. Do not treat UUIDs as secure tokens.

Nano ID (API and constraints)
- Primary functions (JS API extracted):
  - import { nanoid } from 'nanoid'
  - nanoid(size?: number): string  — default size = 21 characters; uses urlAlphabet = "A-Za-z0-9_-"; returns URL-safe ID. Signature: function nanoid(size = 21): string
  - import { customAlphabet } from 'nanoid'
  - customAlphabet(alphabet: string, size: number): () => string  — returns a generator function that accepts optional override size when called. Alphabet must contain <= 256 symbols; size is default output length.
  - import { customRandom, urlAlphabet } from 'nanoid'
  - customRandom(alphabet: string, size: number, randomFn: (size:number)=>Uint8Array|number[]): () => string  — randomFn must accept requested byte count and return an array/Uint8Array of random bytes; used for deterministic or seeded RNGs.
  - Non-secure API: import from 'nanoid/non-secure' provides a Math.random()-based generator (faster but not cryptographically secure).
- Implementation constraints: use secure hardware RNG (crypto module / Web Crypto API); avoid modulo bias (do not use random % alphabetSize naïvely); use rejection sampling or equivalent uniform sampling techniques (algorithm in nanoid source implements buffered uniform sampling to avoid bias).

js-base64 (API surface)
- Module exports (ES / CommonJS): { Base64, encode, decode, atob, btoa, fromUint8Array, toUint8Array, isValid, extendString, extendUint8Array, extendBuiltins }
- Key signatures and behaviours:
  - Base64.encode(input: string, skipPadding?: boolean): string — encode UTF-8 (input coerced to UTF-8) to Base64; skipPadding=true omits trailing '='.
  - Base64.decode(input: string): string — decode Base64 to a UTF-8 string (returns Unicode text); use Base64.atob to decode to raw bytes (byte-to-char semantics matching browser atob).
  - Base64.atob(input: string): string — decode to binary string (one char per byte) consistent with browser atob; suitable for binary-to-bytes conversion.
  - Base64.fromUint8Array(bytes: Uint8Array, skipPadding?: boolean): string — encode byte arrays directly.
  - Base64.toUint8Array(input: string): Uint8Array — decode Base64 to bytes.
  - Base64.isValid(value: any): boolean — tests if value is a valid Base64 input (tolerates whitespace; allows pad omission).
  - Base64.extendString(), extendUint8Array(), extendBuiltins() — optional mutating helpers to attach convenience methods to prototypes.
- Implementation notes: provide separate encode/decode that treat UTF-8 encoding correctly; btoa/atob are preserved for compatibility; decode() returns UTF-8 corrected string and atob() returns raw binary string.

2. SUPPLEMENTARY DETAILS (implementation recipes and algorithmic steps)

Base64 encoding recipe (byte-stream -> base64 string):
- Read input bytes in groups of three: b0, b1, b2.
- Form 24-bit buffer = (b0 << 16) | (b1 << 8) | b2.
- Extract four 6-bit indices: i0 = (buffer >> 18) & 0x3F; i1 = (buffer >> 12) & 0x3F; i2 = (buffer >> 6) & 0x3F; i3 = buffer & 0x3F.
- Map each index to alphabet char.
- If <3 bytes remain: pad buffer with zero bits on right, output appropriate number of chars and append '=' padding to make output block length 4.
- If implementing base64url: replace '+' with '-' and '/' with '_' and optionally omit '=' if spec allows.

Base64 decoding recipe (with padding):
- Remove whitespace (if spec allows) but otherwise reject unexpected characters per RFC 4648 normative guidance.
- Count pad '=' characters (0–2) at end; convert each character to 6-bit value using inverse table; concatenate 6-bit values; drop trailing pad bits (2 bits per '=' encountered) and reconstruct bytes by grouping into 8-bit octets.

Ascii85 encoding recipe (Adobe variant):
- For every 4 input bytes (big-endian), compute v = (b0<<24)|(b1<<16)|(b2<<8)|b3.
- If v == 0, output 'z' (special case) and continue. Else generate 5 digits: for k from 4 down to 0: digit[k] = v % 85; v = Math.floor(v/85); then output chars = digit[k] + 33 in MSB-first order.
- For final partial block (1..3 bytes): pad with NUL bytes (0x00) to 4 bytes before encoding; produce 5 chars, then remove N output chars corresponding to number of pad bytes (e.g., if padded with 3 bytes, output only first 2 chars) as per Adobe spec.

basE91 implementation notes
- Use the official symbol table (91 symbols) and the basE91 packing algorithm: maintain an integer accumulator and bit count, append bits from input bytes to the accumulator, while accumulator has >=13 bits: take lowest 13 bits and map to two output symbols (with conditional branching when value < 8192); otherwise, take 14 bits. A full implementation must follow basE91 pseudo-code exactly for interoperability.

UUID generation rules (practical guidance)
- For v1: gather 60-bit timestamp as 100-ns intervals since 1582-10-15, combine with clock sequence and node ID, set version bits and variant bits as specified. If no stable MAC address is available or privacy is a concern, set node to random 48-bit value and set multicast bit (LSB of first node octet) to 1.
- For v4: fill 128-bit buffer with cryptographic random bytes, set 4-bit version to 4 in the most-significant nibble of the 7th byte, set variant bits to binary 10 in the MSBs of the 9th byte.
- For v7: build output with 48-bit big-endian epoch-ms, set version 7, include extra precision/monotonic bits per implementation, then random bits; ensure variant bits set to 10.

NanoID uniform sampling algorithm (avoid modulo bias):
- Use rejection sampling: produce a random byte from secure RNG; if byte < floor(256 / alphabetSize) * alphabetSize then accept and map to character = alphabet[byte % alphabetSize]; else discard and continue.
- Buffered approach: generate many random bytes at once and iterate to reduce system call overhead; ensure algorithm matches nanoid source ordering for portability if deterministic seeding is used.

3. REFERENCE DETAILS (exact API signatures, configuration options, and effects)

RFC 4648 normative rules (Base64/Base32/Base16):
- Line feeds: "Implementations MUST NOT add line feeds to base-encoded data unless the specification referring to this document explicitly directs base encoders to add line feeds after a specific number of characters." (RFC 4648 §3.1)
- Padding: "Implementations MUST include appropriate pad characters at the end of encoded data unless the specification referring to this document explicitly states otherwise." (RFC 4648 §3.2)
- Non-alphabet characters: "Implementations MUST reject the encoded data if it contains characters outside the base alphabet when interpreting base-encoded data, unless the specification referring to this document explicitly states otherwise." (RFC 4648 §3.3)

Base64 exact alphabet (value->char): see section above (0..63 mapping). Padding char: '='. URL-safe variant: 62='-' 63='_' (pad optional per higher-level spec).

Nano ID API (JS method signatures)
- function nanoid(size?: number): string
  - default size: 21
  - uses urlAlphabet = "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789_-"
  - cryptographic RNG required (Node crypto / WebCrypto). Returns URL-safe string.
- function customAlphabet(alphabet: string, defaultSize: number): (size?: number) => string
  - parameter alphabet: string (max 256 distinct symbols). Returns generator closure.
- function customRandom(alphabet: string, size: number, random: (size:number) => Uint8Array|number[]): () => string
  - The random callback must return an array of uniformly random bytes for the requested size.
- Non-secure export: require 'nanoid/non-secure' yields same API but uses Math.random and is NOT cryptographically secure.
