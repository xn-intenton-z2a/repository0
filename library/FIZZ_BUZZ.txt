1. Problem definition
Given an integer sequence starting at 1 and incrementing by 1, map each integer n to a string according to divisibility rules: if n divisible by 3 map to "Fizz"; if n divisible by 5 map to "Buzz"; if divisible by both 3 and 5 map to "FizzBuzz"; otherwise map to the decimal representation of n.

2. Input domain and constraints
- Inputs: positive integers n >= 1. Typical usage: iterate for n in [1..N] where N is user-specified.
- Constraints: integer arithmetic only; divisibility test is modulo operation. Must handle at least typical 32-bit ranges; for very large N use streaming / generator patterns to avoid memory blow-up.

3. Output rules
- For each input n produce exactly one output string according to mapping above.
- Maintain ordering identical to input sequence.
- No additional side-effects beyond producing sequence items (pure mapping function preferred).

4. Deterministic implementation steps
- For each n compute r3 = n % 3 and r5 = n % 5.
- If r3 == 0 and r5 == 0 return "FizzBuzz".
- Else if r3 == 0 return "Fizz".
- Else if r5 == 0 return "Buzz".
- Else return decimal string of n.
- For languages with zero-based indices, map index i to n = i + 1.

SUPPLEMENTARY DETAILS
- Performance: single pass, O(N) time, O(1) extra memory per item (or O(N) if collecting outputs). Use buffered output when printing large ranges.
- Edge cases: zero and negative inputs are outside canonical definition; define expected behaviour explicitly if needed (e.g., treat non-positive as error).
- Testing: table-driven tests covering multiples of 3 only, 5 only, both, and neither; include boundary cases (1, 2, 3, 4, 5, 15, large N).

REFERENCE DETAILS
- No external APIs; essential implementation pattern is the modulo-based conditional mapping above.
- Implementation pattern for streaming generator: produce items one-by-one using yield or equivalent, computing modulo incrementally if optimizing (update r3=(r3+1)%3, r5=(r5+1)%5) to avoid repeated division in tight loops.
