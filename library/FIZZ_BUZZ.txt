1. PROBLEM SPECIFICATION
Given a positive integer sequence from 1 to N produce a sequence of outputs where each index i (1-based) yields:
- "Fizz" if i divisible by 3 and not by 5
- "Buzz" if i divisible by 5 and not by 3
- "FizzBuzz" if i divisible by both 3 and 5
- The decimal representation of i otherwise
The function must accept an integer N >= 0 and return an array-like sequence of length N with items as strings.

2. RULES AND VARIANTS
- Basic rules: divisibility by 3 and 5 with precedence for both (3*5 -> "FizzBuzz").
- Variant: different divisors or word labels (e.g., 7->"Bang"). Parameters: divisors array and labels array of equal length.
- Variant: inclusive/exclusive ranges (start S, end E) rather than 1..N. Signature may accept start and end.
- Variant: streaming generator instead of full array to reduce memory: yield values one-by-one.
- Variant: predicate-based replacement: list of (predicate, replacement) pairs applied in given order, concatenating matches or using priority.

3. IMPLEMENTATION PRIMITIVES
- Divisibility test: i % d === 0 (JavaScript) or i mod d == 0 (other languages).
- String concatenation for combined matches: let result = ""; for each rule if divisible then result += label; if result === "" then result = String(i);
- Order of rules: when using concatenation, ordering of rules defines output when multiple divisors match.
- Output container: preallocate array of length N and assign by index for O(N) writes and predictable memory layout.
- Generator approach: implement as an iterator object with next() returning { value, done } or using language-specific yield semantics.

4. EDGE CASES AND INPUT VALIDATION
- N == 0 -> return empty array
- N < 0 -> treat as error: throw RangeError or return empty array depending on API contract
- Non-integer N -> coerce via Math.floor or throw TypeError depending on strictness
- Very large N -> document memory usage, recommend generator/streaming API for N > threshold (e.g., >1e6)
- Non-number types in divisors/labels -> validate types and throw TypeError

5. EXAMPLE API PATTERNS AND EXPORTS

Pattern C: generator

Pattern D: object module export (Node/ESM)
  export default function fizzBuzz(...) or module.exports = fizzBuzz

6. PERFORMANCE AND COMPLEXITY
- Time complexity: O(N * R) where R is number of rules (divisors) if checking each rule per i; reduce to O(N) with arithmetic trick only when single rule or precomputed pattern repeats with LCM periodicity.
- Space complexity: O(N) for array output; O(1) for generator/stream implementation aside from output consumer.
- Micro-optimizations: compute divisibility with integer modulus; avoid string concatenation allocations by using conditional branches when R small; precompute labels for least common multiple patterns when R small and repeat period is feasible.

7. SUPPLEMENTARY DETAILS: SPECIFICATIONS AND IMPLEMENTATION NOTES
- Default rule-set: [{divisor:3,label:"Fizz"},{divisor:5,label:"Buzz"}]
- Precedence: when multiple divisors match, labels are concatenated in rule order producing "FizzBuzz" when 3 and 5 both match.
- API contracts:
  - Accept numeric N, throw TypeError if not numeric when strict; otherwise coerce via Number() and floor.
  - Return type must be array of strings for array APIs; iterator for generator APIs.
- Memory recommendation: for N > 1_000_000 prefer generator or paging; document approximate memory per item (string length ~ up to log10(N) + small constant) and therefore estimate total bytes.

8. REFERENCE DETAILS: API SIGNATURES, CONFIGURATION, BEST PRACTICES, TROUBLESHOOTING
- Function signatures (JavaScript/TypeScript):
  - function fizzBuzz(n: number): string[]
    Parameters: n - non-negative integer; Return: string[] length n
  - function fizzBuzzWithRules(n: number, rules: Array<{divisor:number,label:string}>): string[]
    Parameters: rules - array; each divisor must be integer >=1; Return: string[]
  - function* fizzBuzzGenerator(start: number, end: number, rules?): Generator<string>
    Parameters: start <= end; rules optional; yields strings for each i in [start,end]
- Configuration options and effects:
  - rules order affects concatenation order; change order to control label order in combined matches
  - strictMode boolean: if true, throw on invalid inputs; if false, attempt to coerce
  - outputMode: 'array' | 'generator' determines memory behaviour
- Best practices:
  - Validate divisors are integers >=1 and labels are non-empty strings
  - Use generator API for large ranges to avoid memory spikes
  - For deterministic concatenation, document rule order in public API
- Troubleshooting steps:
  - If outputs are missing combined labels, check rule order and concatenation logic
  - If memory spikes, switch to generator or process in pages (chunks) and free references
  - If performance low with many rules, consider computing match bitmask per i using precomputed mod counters
