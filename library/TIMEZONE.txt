1. IANA Time Zone identifiers
- Valid timezone identifiers are strings from the IANA tz database, typically in the form Area/Location (e.g., "America/Los_Angeles", "Europe/London", "Asia/Tokyo").
- Implementations must validate provided tz identifiers against the runtime-supported list (Intl.supportedValuesOf('timeZone') where available) or a bundled IANA list.

2. Using Intl.DateTimeFormat for timezone-aware formatting
- Constructor: new Intl.DateTimeFormat(locale?: string | string[], options?: { timeZone?: string, timeZoneName?: 'short'|'long', hour12?: boolean, year?: 'numeric'|'2-digit', month: 'numeric'|'2-digit'|'long'|'short'|'narrow', day?: 'numeric'|'2-digit', hour?: 'numeric'|'2-digit', minute?: 'numeric'|'2-digit', second?: 'numeric'|'2-digit' })
- Formatting: dtf.format(date) returns a locale-formatted string representing the Date in the specified timeZone.
- Use dtf.formatToParts(date) to get individual components (year, month, day, hour, minute, second, timeZoneName) for reconstructing local wall-clock components instead of parsing formatted strings.

3. Converting instants to wall-clock components
- To compute the scheduler's firing behavior in a given IANA timezone, convert the reference UTC instant to the target timezone's wall-clock fields using Intl.DateTimeFormat(formatToParts) or Temporal (if available).
- Example algorithm: let parts = dtf.formatToParts(utcDate); extract numeric values for year, month, day, hour, minute, second; use these as the local wall-clock representation for matching against cron field bitsets.
- When generating the next fire time, prefer to compute candidate local times and convert back to UTC instants for storage and triggering using Date.UTC with components adjusted for the timezone offset.

4. DST semantics and edge cases
- Missing local times (spring forward): local times in the gap do not map to an instant. Scheduler options: skip (do not fire), shift forward to next valid instant, or normalize to a canonical rule (e.g., use UTC mapping). Document chosen policy and make it configurable.
- Ambiguous local times (fall back): one local time maps to two instants. Scheduler options: pick first occurrence, pick second occurrence, or fire twice. Persist policy.
- When computing next occurrences near DST boundaries, compute using timezone-aware conversion rather than naive arithmetic on local fields to avoid off-by-one-hour errors.

5. Integration patterns for schedulers
- Validate tz at job creation time; reject invalid IANA identifiers.
- Persist both user-specified timezone and the last-fired UTC instant to resume reliably.
- When listing upcoming runs, present wall-clock times in the job timezone and also include corresponding UTC ISO timestamps.
- Tests: cover DST forward, DST backward, leap-year Feb-29, end-of-month rollovers, timezone offsets with non-hour increments (e.g., Asia/Kolkata +05:30).

SUPPLEMENTARY DETAILS

Implementation recommendations
- Use Intl.DateTimeFormat#formatToParts when Temporal is unavailable; prefer Temporal.PlainDateTime/Instant when available for clearer semantics.
- Maintain a small cached map of timeZone -> Intl.DateTimeFormat instances for performance per locale/option combination.
- For environments without full IANA support, ship a minimal tz validation table and fall back to UTC when invalid.
