NORMALISED EXTRACT (direct technical content)
1. Time Zone Database (tzdb) overview
- The tz database (zoneinfo) contains historical and current local time data for locations worldwide, including UTC offsets and DST transition rules.
- Maintained as a canonical source of IANA time zone identifiers used by most OS, runtime libraries, and timezone-aware date libraries.

SUPPLEMENTARY DETAILS (uptake for cron-engine)
- Use tzdb canonical zone names (Region/City) in scheduler configuration and in user-facing UI. Example: Europe/London, America/New_York.
- For deterministic scheduling around DST, store schedules in either:
  - IANA zone-aware form and compute next runs via tz-aware library (recommended), or
  - convert schedule to UTC for storage and compute next run in UTC (less human-friendly around DST boundaries).
- Keep tzdb up-to-date: cron engines must offer a mechanism to update tzdb (pull new tzdata releases) and re-evaluate future schedule computations when tz rules change.

REFERENCE DETAILS (implementation specifics)
- Recommended artifact for programmatic installs: tzdata2026a.tar.gz (data distribution) or use OS package management that updates zoneinfo in /usr/share/zoneinfo.
- For Node/JS: rely on runtime ICU + Intl or libraries (Luxon, moment-timezone) that are built against a tzdb; ensure underlying tzdb is updated when IANA releases new data.
- When computing next occurrence: use library that accepts IANA tz and computes DST-aware offsets; do not attempt to hardcode DST transitions.
