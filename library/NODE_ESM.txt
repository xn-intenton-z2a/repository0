1. Enabling ESM in Node.js
- Mark module as ESM by either:
  - File extension .mjs; or
  - package.json field "type": "module" to treat .js as ESM; or
  - CLI flag --input-type=module for evaluation contexts
- Conversely, use .cjs or "type": "commonjs" to force CommonJS.

2. Import specifier types and resolution rules
- Relative specifiers require file extension and are resolved by URL semantics; directory indexes must be fully specified.
- Bare specifiers resolved via Node's package resolution algorithm; packages with package.json exports field restrict accessible internal paths to those listed.
- Absolute file: URLs supported; percent-encode special characters.
- Mandatory file extension: when using import with relative/absolute specifiers, file extension is required in Node.js to conform to browser-like semantics.
- data: imports supported for types: text/javascript (ESM), application/json (JSON modules), application/wasm (WASM).

3. import.meta properties and APIs
- import.meta.url: absolute file: URL string of current module.
- import.meta.filename: absolute path with symlinks resolved (file: modules only).
- import.meta.dirname: directory name string of current module (file: modules only).
- import.meta.main: boolean true when module is entry point of process.
- import.meta.resolve(specifier: string, parentURL?: string|URL) -> string absolute URL of resolved specifier; synchronous, may perform filesystem operations; caveats for loaders.

4. Interoperability with CommonJS
- Importing CommonJS from ESM yields a module namespace object with default export set to module.exports and possible named exports provided via static analysis.
- module.createRequire(filename) can construct a require() inside ESM to load CommonJS or other assets.
- Differences: no __filename or __dirname globals; use import.meta.filename and import.meta.dirname instead; addons not supported via ESM import (use createRequire).

SUPPLEMENTARY DETAILS

Technical specifications and implementation details:
- Node.js versions: features stabilized across v12.x through v24+; import.meta.filename and dirname present in v21.2.0+ and stable by v24.0.0.
- Module caching: identity keyed by resolved URL including query/fragment; different specifiers load separate module instances.
- Built-in 'node:' URL scheme: import builtin modules via 'node:fs' or 'node:fs/promises' to reference core APIs explicitly.

REFERENCE DETAILS

API signatures and behaviors (Node.js ESM):
- import(specifier: string) -> Promise<ModuleNamespace>
- import.meta.resolve(specifier: string, parentURL?: string|URL) -> string (absolute URL)
- module.createRequire(filename: string) -> Function require
- syncBuiltinESMExports() -> void (module.syncBuiltinESMExports) updates named exports of builtins after mutation

Implementation patterns and troubleshooting:
- If "Cannot find module" for bare specifier: ensure package.json "exports" allows access or set up package resolution; for browser use import maps.
- If top-level await not working: ensure Node version supports top-level await (stable in newer Node versions) and module is ESM.
- If JSON import fails: import with attribute type 'json' or use import assertions: import data from './file.json' with { type: 'json' }
