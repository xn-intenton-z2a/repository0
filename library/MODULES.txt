1. Module formats: ESM vs CommonJS
- ESM (ECMAScript modules): import/export syntax, static structure, supports live bindings, static analysis, tree-shaking.
- CommonJS: require(), module.exports, dynamic loading, mutable export copy (by value), executed per-require caching semantics.

2. Enabling ESM in Node.js
- Use file extension .mjs OR set package.json "type": "module" to interpret .js as ESM.
- When "type": "module" is present, .cjs files are CommonJS and .js are ESM.
- For packages intended for both, provide dual package: include both ESM entry in "exports" field and CommonJS in "main" or conditional exports.

3. Import syntax and forms
- Static import: import defaultExport from 'module'; import { named } from 'module'; import * as ns from 'module';
- Dynamic import: import('module').then(m => ... ) returns a Promise resolving to module namespace object.
- Import assertions: import data from './file.json' assert { type: 'json' } when importing JSON modules (Node.js supports import assertions).

4. Module resolution algorithm (Node.js specifics)
- Resolution follows spec: bare specifiers resolved via package exports/exports map or node_modules lookup; relative/absolute specifiers resolved to file paths with extensions appended per allowed list and directory package checks.
- package.json "exports" field controls public entry points and subpath exports; if present, it restricts direct path access unless enumerated.

5. Interoperability patterns
- CommonJS consumer importing ESM: use dynamic import() to load ESM; or use createRequire and import the transpiled CommonJS build.
- ESM importing CommonJS: import default from CommonJS module yields namespace object where default is module.exports; Node applies default interop so import pkg from 'cjs' may give { default: module.exports } unless configured.
- Use Module.createRequire(import.meta.url) to obtain require in ESM contexts.

6. Top-level features and constraints
- Top-level await: allowed in ESM module contexts, suspends module instantiation until awaited promises resolve.
- __filename and __dirname are not defined in ESM; use import.meta.url and URL APIs to derive file path.
- Circular dependencies: rely on live bindings for ESM; CommonJS may return partially initialised exports.

SUPPLEMENTARY DETAILS
- Packaging: prefer specifying both "main" (CommonJS) and "exports" with conditional exports for ESM consumers.
- Performance: static ESM allows more optimizations; dynamic import incurs Promise resolution cost.

REFERENCE DETAILS
- Node package.json relevant fields: "type", "main", "exports", "module" (convention), "imports".
- Module.createRequire(moduleUrlString) -> returns CommonJS require function usable in ESM.
- import(moduleSpecifier) -> Promise<ModuleNamespaceObject>
- import.meta.url -> string URL of current module file.
